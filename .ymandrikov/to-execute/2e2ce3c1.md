# Overengineering Analysis: Yarn Lockfile Identity Plan (ef9a6b53cbd8.md)

**Date:** 2026-07-14

**Subject:** `.ymandrikov/to-execute/ef9a6b53cbd8.md`

## Verdict

The core design is sound. The bug is real, the ownership call is right (fix belongs in the Yarn loader, not `splitPackage`), and a pure parser module is a reasonable shape. But roughly 40% of the plan is ceremony. The current loader is 146 lines; the plan swells it to 500+ across two files. The fix needs maybe half that.

## Sound — keep

- Splitting grouped keys on `', '`, the scope-aware first-`@` separator, and npm alias target extraction. This *is* the fix.
- Not widening `splitPackage`. Grouped keys, quoting, aliases, and locators are Yarn serialization concepts; the shared helper's single-coordinate contract should stay intact.
- Offline, deterministic fixtures and warn-and-skip on parse failure.

## Overengineered — cut

### 1. Internal brand explosion (biggest offender)

The plan defines seven brands: `YarnDescriptor`, `YarnIdent`, `YarnReference`, `YarnTargetName`, `YarnRequestedName`, `YarnResolvedName`, `YarnAggregationKey`. The codebase brands public domain values only (`FilePath`, `DependencyName`, etc. in `common/types.ts:24-37`) — never intermediate parse strings inside a single module. The consequence is six one-line cast helpers (`descriptorType`, `identType`, `referenceType`, `requestedType`, `targetType`, `resolvedType`) — about 30 lines of pure noise that do nothing at runtime or compile time worth having.

**Fix:** keep at most two brands on the exported API (`YarnRequestedName`, `YarnResolvedName`); use plain strings internally. Note the constraint is self-imposed by the plan's own Global Constraints section, not by any codebase convention.

### 2. `YarnText` missing/content wrapper

`{ missing: true } | { missing: false; content: string }` replaces `string | undefined`. TypeScript already narrows optionals — the current code does `dep.resolved?.includes(...)` without issue. The union adds no third state and no information. Also a self-imposed constraint ("represent missing values with discriminated unions").

**Fix:** use `string | undefined`.

### 3. Failure-reason taxonomy for machine-generated input

Lockfiles are written by Yarn, not humans. "Unmatched quotes", "Embedded quotes", "Empty descriptor", and `Descriptor N:` prefixes defend against inputs Yarn never emits. Tests then assert exact reason strings (`'Descriptor 2: No reference'`), coupling the suite to message wording.

**Fix:** one generic "unparseable key" warning with source path and raw key. Drop reason-string assertions.

### 4. Berry locator fallback ladder

"Missing or invalid locator → agreed-target fallback → targets disagree → fail" — but Berry always writes `resolution` for package entries. The whole ladder (final-state invariants 7–8, four test cases, a dedicated risk section) guards a state that does not occur. The classic path needs target-agreement logic anyway.

**Fix:** Berry = "use the locator; if absent, treat as classic." One sentence, not a state machine.

### 5. Task 2 Step 0 is a no-op experiment

A manual out-of-repo Yarn 1 install whose result changes nothing — the plan itself says "Either result leaves aggregation mandatory." An experiment that gates no decision is ritual.

**Fix:** cut it, or let its outcome decide whether Task 2 ships at all.

### 6. Aggregation's "mandatory" framing is inconsistent

The plan admits `calculateVersionDiff` ignores source paths, so identical cross-lockfile duplicates survive downstream anyway. Within-lockfile dedup is treated as a sacred contract while the same duplicate one directory over is tolerated. The aggregation code itself is cheap (~40 lines) and the directness union is real value — fine to keep — but:

- the "first-occurrence position" invariant plus six dedicated tests is excess for output whose order has no stated consumer;
- the `JSON.stringify` aggregation key includes a `type` field that is the hardcoded literal `'npm'` for every Yarn dependency — a 2-tuple dressed as a 3-tuple.

**Fix:** keep aggregation and the directness union; drop the position invariant and the constant `type` from the key.

### 7. TDD theater

Steps say "write failing tests," then paste the exact implementation in the same document three steps later. Tests derived from an implementation given in the same file are transcription, not test-first design.

## Minor smells

- `resolveYarnIdentity` ends with an unreachable `return { invalid: true, reason: 'Descriptor group is empty' }` after a `for (let target of targets)` loop used to extract a single Set element — a contortion to satisfy the type checker, symptomatic of the ceremony pressure elsewhere.
- `requested: ['pkg', 'pkg']` keeps duplicates in the requested list, which aggregation later Set-dedupes. Pick one representation.
- 18 final-state invariants, 10 rejected alternatives, and a domain-model table for a lockfile-key parser. The plan document is ~1800 lines for ~250 lines of code.

## Recommended shape

Ship Task 1 with brands trimmed to the exported API, `string | undefined` instead of `YarnText`, and one generic parse warning. Keep aggregation but drop the position invariant and the constant `type` from the key. Cut Task 2 Step 0. Same correctness, roughly half the code and test surface.
