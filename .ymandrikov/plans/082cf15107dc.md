This document merges the two competing final plans. Every point they

## Decision

The Yarn loader will model requested and resolved package identities
separately. Yarn-specific parsing will move out of the shared `splitPackage`
helper and into a new pure module, `server/loader/versions/yarn-identity.ts`.

For every valid Yarn lock entry, the loader will derive:

- every requested package name represented by the entry;
- one canonical resolved package name;
- one effective version;
- the existing fetch resolution, when present.

The public dependency keeps the existing shape:

```text
Dependency.name   = canonical resolved package name
Dependency.direct = package.json contains any requested package name
```

Entries whose identity cannot be established are excluded **with a warning**
naming the source file and the raw key, via the existing `warn()` helper
(`server/cli/print.ts:20`, already used from loader code at
`server/loader/github.ts:42`). Exclusion is deterministic; it is never
silent.

Cross-entry coalescing (one dependency per equal public resolved identity per
lockfile) is **gated on evidence**: stage 3 starts with a verification task
that checks both known potential sources of duplicate public identities. If
neither is confirmed with a real fixture, stage 3 is dropped.

Delivery is staged:

1. Ship a narrow grouped-key fix for issue #55.
2. Introduce structural descriptor and locator parsing with explicit identity
   selection.
3. Verify the duplicate-identity claims; if confirmed, add cross-entry
   coalescing.

Stage 1 ships independently. Stages 2 and 3 form the long-term change and are
implemented as separate, reviewable commits in one pull request, released
together.

## Problem

Yarn Berry and Yarn 1 can serialize multiple descriptors that resolve to the
same package as one comma-separated lockfile key:

```text
caniuse-lite@npm:^1.0.30001702, caniuse-lite@npm:^1.0.30001746
```

The current Yarn parsers pass that entire key to `splitPackage`, which
accepts one coordinate. Its last-`@` search crosses the descriptor boundary
and returns this package name:

```text
caniuse-lite@^1.0.30001702, caniuse-lite
```

The malformed name causes two visible failures:

1. Direct-dependency detection fails because it does not match `caniuse-lite`
   from `package.json`, so a genuinely direct dependency is shown as
   indirect.
2. Npm diff loading includes the malformed name in a registry URL
   (`https://registry.npmjs.org/caniuse-lite@^1.0.30001702, caniuse-lite/-/...`)
   and receives a 404, surfacing as `Error: Failed to download tarball`.

The defect belongs to the Yarn format boundary. A grouped Yarn key is a
collection of descriptors, not a package coordinate. Changing `splitPackage`
or replacing every `@npm:` occurrence would alter the malformed string
without restoring the descriptor boundary.

The issue is reported against Yarn Berry, but Yarn 1 uses the same merged-key
serialization through the same parser, so both formats are in scope.

## Root cause and evidence

For the merged key above, `splitPackage`
(`server/loader/versions/common.ts:36`) returns:

```text
name:    caniuse-lite@^1.0.30001702, caniuse-lite
version: npm:^1.0.30001746
```

Both Yarn call sites pass merged keys unsplit:

| Parser | Location |
|---|---|
| `parseYarn1Lock` | `server/loader/versions/yarn.ts:56` |
| `parseYarnBerryLock` | `server/loader/versions/yarn.ts:87` |

Facts verified against the code that constrain the fix:

- Only `name` from `splitPackage` is consumed by the Yarn loaders; the
  version always comes from the entry body (`version:` field /
  `version "..."` line). The garbage `version` return value is harmless on
  the Yarn path.
- `pkg.replace('@npm:', '@')` (`common.ts:44`) replaces only the first
  occurrence, but a valid single coordinate contains at most one `@npm:`, so
  this is not a separate defect once descriptors are split. `replaceAll`
  would change the malformed output without restoring the missing boundary.
- The Berry case is a regression from commit `9797117` (`Fix scoped package
  support`), which replaced Berry-specific name extraction with the shared
  `splitPackage` call. The Yarn 1 case is a pre-existing instance of the
  same bug.
- Both parsers already exclude entries without a usable version
  (`yarn.ts:53`, `yarn.ts:84`); Berry already excludes `__metadata` and
  `@workspace:` keys (`yarn.ts:75`). The failure-semantics table below
  builds on these existing exclusions.
- After stage 2, the only remaining `splitPackage` consumer is
  `server/loader/versions/pnpm.ts:38`, whose keys arrive from parsed YAML
  (already unquoted, never containing `@npm:`).

## Why existing tests miss it

`server/test/yarn-berry.test.ts` and `server/test/yarn1.test.ts` are
integration tests: they install a single package with the real Yarn binary
against the live registry. A single direct dependency never produces a
merged key, so the shape is structurally unreachable in current fixtures,
and live-registry tests are too slow and rate-limited to force it (the
issue reporter hit rate limits attempting exactly that).

The missing coverage belongs at the `yarn.load()` boundary with in-memory
`LoadedFile` string fixtures: deterministic, offline, and covering format
detection, parsing, branded `Dependency` construction, and direct-dependency
detection in one pass.

## Domain model

| Term | Meaning |
|---|---|
| Descriptor | A requested package ident and its range or protocol reference |
| Requested name | The outer package ident under which a dependency was requested |
| Target name | The package ident named by descriptor semantics; differs from the requested name for an npm alias |
| Locator | A resolved package ident and the reference identifying its source |
| Resolved name | The package ident represented by the selected locator or agreed descriptor target |
| Lock entry | One serialized resolution and every descriptor mapped to it |
| Effective version | The entry version after preserving the loader's existing git-resolution behavior |
| Public dependency identity | Resolved name, effective version, and dependency type |

For an ordinary dependency, all three names agree:

```text
requested: caniuse-lite
target:    caniuse-lite
resolved:  caniuse-lite
```

For an npm alias, the requested name differs:

```text
package.json: "my-alias": "npm:lodash@^4.0.0"
descriptor:   my-alias@npm:lodash@^4.0.0
locator:      lodash@npm:4.17.21

requested: my-alias
target:    lodash
resolved:  lodash
```

The requested name determines directness. The resolved name determines
change identity, npm tarball loading, repository lookup, and display. For an
aliased dependency the UI shows `lodash`, not `my-alias` — the right
default, since the changed code is lodash's. If alias display becomes a
requirement later, it needs a second field (e.g. `declaredAs`), not a
different `name`.

The distinction remains internal to the Yarn loader. This change does not
add an alias or locator field to `Dependency` or `Change`.

## Invariants

1. One lock entry produces at most one parsed resolution.
2. Descriptor count never determines dependency count.
3. Equal public dependency identities emit once per lockfile
   *(stage 3; drops to per-entry scope if the stage 3 gate fails)*.
4. Coalescing unions every requested name represented by its members
   *(stage 3, same gate)*.
5. `Dependency.name` is the canonical resolved name used by downstream npm
   loading.
6. `Dependency.direct` is derived from requested names, never from the
   resolved name alone.
7. Descriptor order never selects package identity
   *(stage 1 knowingly and temporarily violates this for alias keys, which
   are equally broken today; on same-ident merged keys — every input whose
   behavior stage 1 changes — first-descriptor and agreed-target produce
   identical output)*.
8. Nested `@` characters in references never affect the outer package ident.
9. Ambiguous identity is excluded instead of guessed, and every such
   exclusion emits a warning.
10. Yarn serialization rules do not change the contract of `splitPackage`
    for pnpm input or of any non-Yarn loader.
11. Git resolutions remain distinct when their effective versions differ.
12. Non-git locators with the same public resolved identity intentionally
    coalesce because the current public model cannot represent a
    locator-level distinction *(stage 3, same gate)*.

If locator-level differences must become visible later, the public
dependency model and diff identity must be expanded explicitly. They must
not be smuggled into this loader's aggregation key.

## Architecture

The implementation has three responsibilities.

### Lockfile format parsing

`server/loader/versions/yarn.ts` continues to:

- detect Yarn Berry or Yarn 1;
- parse raw lock entries;
- exclude metadata and workspaces;
- read manifests and direct dependency names;
- preserve existing git-version handling (`yarn.ts:124-129`);
- construct branded `Dependency` values;
- emit warnings for excluded entries.

Format parsing produces a discriminated raw-entry union for Berry and
Yarn 1. It preserves the grouped key, version, and format-specific
resolution without choosing a package identity while scanning the lockfile.

### Yarn identity parsing

A new pure `server/loader/versions/yarn-identity.ts` module owns:

- grouped-key splitting and Yarn 1 quote handling;
- outer descriptor parsing;
- npm alias target parsing;
- Berry locator parsing;
- canonical-name selection;
- ambiguity detection.

Type rigor is targeted at the swap hazard that caused this bug class:

- `RequestedName`, `TargetName`, and `ResolvedName` are distinct branded
  string types (following the existing repo idiom of `FilePath`,
  `ServerURL`, `DependencyName`), so requested and resolved identities
  cannot be substituted accidentally.
- Parsing and selection return discriminated success-or-failure values.
  They do not use optional content to represent a failed parse, and they
  never use unchecked casts or `as any`.
- References, descriptors, and locators are plain structs; branding them
  adds ceremony without guarding a real confusion in a single-consumer
  module.

The module has no filesystem, process, registry, or diff-loading
dependency. It exports the specific functions and types needed by `yarn.ts`
and its tests as named exports (the repo has no test-only-export pattern,
and this module must not invent one). TypeScript imports include the `.ts`
extension and no default export is introduced.

### Resolution aggregation (stage 3, gated)

The Yarn loader aggregates valid parsed resolutions separately for each
source lockfile. Within that scope, the key contains:

- canonical resolved name;
- effective version;
- dependency type.

Each aggregate unions requested names. Directness is calculated after the
union so a direct alias and a transitive target, or the reverse, produce one
direct dependency.

Aggregation happens after existing git handling chooses the effective
version. Different git commits therefore remain distinct. Equal non-git
public identities coalesce even when their raw locator references differ.
Scoping the aggregation to one source lockfile preserves monorepo boundaries
while the key matches the identity visible to `calculateVersionDiff` and npm
loading.

This layer is built only if the stage 3 verification confirms a real
duplicate-identity shape (see "Staged delivery").

## Structural parsing

### Outer coordinates

A Yarn descriptor or locator has an outer `ident@reference` structure. A
scoped ident can contain only its leading scope marker before the separator.
The outer separator is therefore the first `@` after the scope, not the
final `@` in the complete string. Input without a separator is an invalid
coordinate, not an ident-only success:

```ts
function splitOuter(input: string): ParseResult<YarnCoordinate> {
  let at = input.indexOf('@', input.startsWith('@') ? 1 : 0)
  if (at === -1) return { invalid: true }
  return {
    invalid: false,
    value: {
      ident: input.slice(0, at),
      reference: input.slice(at + 1)
    }
  }
}
```

The real implementation must validate that the ident and reference are
usable. The example only establishes the separator rule and the failure
shape.

This strategy preserves the outer ident for references containing nested
`@` characters, including:

- `npm:^1.0.0`;
- `npm:lodash@^4.0.0` (alias);
- `git@github.com:ai/nanoid.git#commit=abc`;
- `patch:lodash@npm%3A4.17.21#...`;
- `portal:` and `workspace:` references.

It is the same strategy as Yarn's own `structUtils.parseDescriptor` /
`parseLocator`.

### Grouped keys

Berry grouped keys are split on Yarn's exact `', '` serialization delimiter
after YAML parsing; npm semver ranges cannot contain a comma, so the split
is unambiguous. Yarn 1 headers use the same descriptor delimiter but can
quote each descriptor separately
(`"@babel/core@^7.0.0", "@babel/core@^7.20.0":`), so quotes are stripped per
descriptor after the split.

The implementation must reject empty members, unmatched quotes, and members
without a valid requested ident. It must not silently drop one malformed
member and retain incomplete directness information for the rest of the
entry — a partially parsed key excludes the whole entry, with a warning.

### Descriptor targets

Every descriptor yields:

- the outer requested name;
- a target name;
- the remaining reference.

For an ordinary descriptor, the target name equals the requested name. For
an npm alias reference of the form `npm:<target-ident>@<selector>`, the
target name is `<target-ident>`. Scoped alias targets use the same
outer-coordinate rule (`npm:@scope/pkg@^1.0.0` → target `@scope/pkg`).
A plain npm range (`npm:^1.0.0`) contains no inner separator, so the inner
parse fails and the descriptor is not an alias — this non-alias fall-through
must be handled explicitly, not inferred from an empty-string convention.

Alias detection must be structural. It must not infer a target from
arbitrary registry URLs or assume that every nested `@` belongs to an npm
alias. Non-`npm:` references (git, patch, portal, plain Yarn 1 ranges) keep
the descriptor's own ident as the target.

### Berry locators

When a Berry `resolution` locator parses successfully, its outer ident is
the canonical resolved name. The locator is authoritative: it names the
package the entry actually represents, and outer-ident parsing is immune to
nested `@` in the reference. Descriptor names remain relevant for directness
and fallback only.

If the locator is missing or invalid, fallback succeeds only when:

- every descriptor parses completely; and
- every descriptor target agrees on one name.

If targets disagree, the entry is ambiguous and is excluded with a warning.

### Yarn 1 identities

Yarn 1's `resolved` field is a fetch URL rather than a Yarn locator. Custom
registries make URL-path identity inference unreliable, so it is not used as
a package-name source.

The canonical resolved name comes from descriptor targets. Every target must
agree. This supports ordinary and npm-alias descriptors without depending on
a registry URL layout. Disagreement excludes the entry with a warning.

If Yarn 1 serializes an alias and its target as separate entries (stage 3
verification), each entry is parsed independently and the aggregation phase
coalesces their equal public resolved identities.

## Failure semantics

Correctness takes precedence over inventing a plausible identity, but no
new exclusion class is silent: multiocular is a dependency-review tool, and
silently hiding an entry hides a real change from the reviewer — worse than
a mislabeled one. Every warning names the source file and the raw lockfile
key.

| Condition | Behavior |
|---|---|
| File-level syntax error | Preserve the existing file-level parse failure |
| Metadata or workspace entry | Exclude (existing behavior, `yarn.ts:75`) |
| Entry without a usable version | Exclude (existing behavior, `yarn.ts:53`, `yarn.ts:84`) |
| Malformed grouped key (empty member, unmatched quotes) | Exclude entry + `warn()` |
| Any descriptor in a key fails to parse | Exclude entry + `warn()` — directness would be incomplete |
| Descriptor without a requested or target name | Exclude entry + `warn()` |
| Valid Berry locator | Use its outer ident as the resolved name |
| Missing or invalid Berry locator, all descriptor targets agree | Use the agreed target |
| Missing or invalid Berry locator, targets disagree | Exclude entry + `warn()` |
| Yarn 1 descriptor targets agree | Use the agreed target |
| Yarn 1 descriptor targets disagree | Exclude entry + `warn()` |

Entry exclusion remains local to the invalid entry. A malformed entry does
not invalidate unrelated valid entries unless the format parser already
treats the file as syntactically invalid.

Known limit: `warn()` writes to stderr. CLI users see it; web-UI users only
see it in the server log. Structured loader diagnostics surfaced in the UI
are a separate product change if users need them.

## Staged delivery

### Stage 1 — issue #55 hotfix

Add a Yarn-local helper that selects the first serialized descriptor using
the exact `', '` delimiter before passing that single descriptor to
`splitPackage`. Both Yarn parsers use the helper. Using the exact delimiter
mirrors the stage 2 `parseMergedKey` rule, so stage 1 is a degenerate form
of the long-term parser.

This stage fixes:

- the reported Berry `caniuse-lite` key;
- grouped scoped Berry keys;
- grouped Yarn 1 keys, including per-descriptor quotes;
- directness when grouped descriptors share the same requested name.

This stage intentionally does not fix aliases (they mis-parse even unmerged
today) and does not change:

- `splitPackage`;
- pnpm parsing;
- public types or APIs;
- dependency declarations;
- `eslint.config.ts`.

Stage 1 is released independently so the reported registry 404 is not
blocked by the identity refactor. The helper is temporary and must be
removed by stage 2.

### Stage 2 — structural identity parser

Introduce `yarn-identity.ts`, change both format parsers to retain grouped
keys, and select canonical identities using the rules above.

Stage 2 delivers:

- all requested names from grouped keys;
- structural outer-ident parsing;
- Berry locator authority;
- npm alias target parsing for Berry and Yarn 1;
- consensus-based fallback;
- deterministic, warned exclusion of ambiguous entries;
- removal of the `splitPackage` import from `yarn.ts`;
- conditional `splitPackage` cleanup: once pnpm fixtures confirm that pnpm
  never feeds quoted or `@npm:`-bearing input, remove the Yarn quote branch
  and the `@npm:` replace (`common.ts:41-44`) and fix the stale
  `"@types/node@npm:22.0.0"` doc example; otherwise leave them.

At this stage one valid parsed resolution still corresponds to one lock
entry. Tests establish the per-entry identity behavior before cardinality
changes.

### Stage 3 — public-identity aggregation (gated on verification)

Stage 3 starts with a verification task covering both known potential
sources of duplicate public identities:

1. **Yarn 1 alias-split claim:** generate a real Yarn 1 lockfile containing
   `lodash` plus `my-alias: npm:lodash@<same version>` and inspect whether
   the alias and its target serialize as separate entries.
2. **Berry builtin-patch claim:** generate a real Berry lockfile for a
   project depending on `typescript` and inspect whether the lockfile holds
   both a `typescript@npm:...` entry and a
   `typescript@patch:typescript@npm%3A...#builtin<compat/typescript>` entry
   with the same resolved name, version, and type.

Tooling note: neither `yarn` nor `corepack` is on PATH in this environment
(Node 26 unbundled corepack). Generate fixtures via `npm exec yarn@1` /
`npm exec yarn` or a one-off global install; the resulting lockfiles are
committed as in-memory string fixtures, so no runtime Yarn dependency is
introduced.

If either shape is confirmed, implement aggregation (per-lockfile, after
effective-version calculation, union requested names before directness) and
emit one dependency per public resolved identity. `calculateVersionDiff`
(`server/loader/versions.ts`) has no dedup by id, so duplicate identities
would otherwise produce duplicate `Change` records.

If neither shape is confirmed, stage 3 is dropped and invariants 3, 4, and
12 relax to per-entry scope.

Stage 3 belongs in the same pull request as stage 2 but in a separate
commit with its own focused tests, so the cardinality change is reviewable
in isolation.

## Data flow

```text
yarn.lock
  -> detect Berry or Yarn 1
  -> parse raw lock entries
  -> split grouped keys into descriptors
  -> parse requested names and descriptor targets
  -> parse the Berry locator when available
  -> select one canonical resolved name per entry, or exclude + warn
  -> preserve existing git effective-version handling
  -> [stage 3] coalesce equal public dependency identities per lockfile
  -> [stage 3] union requested names
  -> calculate directness from requested names
  -> emit one branded Dependency per entry (per aggregate after stage 3)
  -> calculateVersionDiff compares resolved names and versions
  -> npm loading receives the resolved registry package name
```

integration tests remain unchanged. Loader-boundary tests call `yarn.load()`

and the array length. No grouped-key behavior is tested through

## Acceptance criteria

### Stage 1

- The issue fixture emits exactly one direct `caniuse-lite@1.0.30001750`
  dependency.
- Grouped scoped keys work in Berry and Yarn 1.
- Single-descriptor behavior remains unchanged.
- `splitPackage`, pnpm, public APIs, dependencies, and `eslint.config.ts`
  remain unchanged.

### Stage 2

- `Dependency.name` always contains the canonical resolved package name.
- Berry aliases use the locator ident and remain direct through a declared
  alias.
- Yarn 1 aliases derive their target from descriptor syntax, never a fetch
  URL.
- Descriptor order cannot change canonical identity.
- Missing-locator fallback requires complete target agreement.
- Ambiguous entries are excluded with a `warn()` naming file and key —
  never emitted under an arbitrary name, never dropped silently.
- Nested locator references cannot add `@` characters to the resolved name
  after the scope marker.
- `yarn.ts` no longer imports `splitPackage`.
- `splitPackage` keeps its behavior for all pnpm-shaped input; its dead
  Yarn branches are removed only if pnpm fixtures confirm they are
  unreachable.
- `Dependency`, `Change`, `calculateVersionDiff`, npm loading, and pnpm
  loading keep their current contracts.
- New coverage is deterministic and offline.

### Stage 3 (only if the gate confirms)

- Equal public identities within one lockfile emit once and preserve all
  requested names; directness considers every requested name in the
  aggregate.
- Different git effective versions do not coalesce.
- The confirmed duplicate-identity fixture(s) are committed as regression
  tests.

### All stages

- Verification passes in this order:
  `./scripts/format.sh`, `pnpm test:types`, `pnpm test`.

## Risks and controls

### Stage 1 remains intentionally incomplete

Selecting the first descriptor repairs grouped entries whose descriptors
share one requested name, but it does not establish a general identity
rule. The helper is temporary and must be removed by stage 2. On every
input stage 1 changes (same-ident merged keys), first-descriptor and
agreed-target output is identical, so stage 1 violates no stage 2 invariant
on any input it affects.

### Exclusion can hide an ambiguous dependency

The long-term parser skips an entry rather than emitting a guessed name.
`warn()` makes each skip visible, but only on stderr — the web UI does not
surface it. Tests must cover every exclusion condition so valid common
protocols are not rejected accidentally. Structured UI diagnostics are a
separate product change if needed.

### Aggregation changes cardinality

Coalescing can reduce duplicate dependencies that the current loader
exposes. The stage 3 gate ensures the layer is built against a confirmed
real shape, not a synthetic one; the separate commit isolates review and
tests. The aggregation key mirrors the identity currently visible to
downstream comparison.

### The gate itself can be wrong

If both verification fixtures come back negative, stage 3 is dropped — but
a future Yarn version could still introduce duplicate-identity
serialization. The pure module keeps aggregation cheap to add later; the
decision log records why it was skipped.

### Yarn syntax can evolve

Protocol assumptions live in one pure module with explicit protocol-zoo
tests. Unknown but structurally valid references retain their outer ident.
Syntax that cannot establish requested and resolved identity is rejected
deterministically, with a warning.

## Decision log

Resolutions from the 2026-07-14 decision review that merged the two
predecessor plans:

1. **Warned exclusion over silent exclusion.** A review tool must not hide
   a real change without a trace. `warn()` exists (`print.ts:20`), has
   loader precedent (`github.ts:42`), and costs nothing. Known stderr
   limitation accepted; UI diagnostics deferred.
2. **Coalescing gated on widened verification.** The Yarn 1 alias-split
   claim was load-bearing and unverified in both predecessors; the Berry
   builtin-patch dual-entry shape is a second unverified duplicate source
   neither document checked. Verify both with real fixtures; build
   aggregation only on confirmed evidence.
3. **Three stages.** Parser rewrite and cardinality change are reviewed and
   tested separately (separate commits, one PR), with stage 3 conditional
   on the gate.
4. **Separate pure module `yarn-identity.ts`.** The protocol zoo is
   unreachable through `yarn.load()` fixtures alone, and the repo has no
   test-only-export pattern; named exports from a pure module avoid
   inventing one. `yarn.ts` stays format orchestration.
5. **Targeted type rigor.** Brand only `RequestedName` / `TargetName` /
   `ResolvedName` — the swap that caused this bug class — plus
   discriminated `ParseResult` and no-`@`-is-invalid. Full six-concept
   branding rejected as ceremony for a single-consumer module.
6. **Conditional, widened `splitPackage` cleanup.** Post-stage-2 consumers
   are pnpm-only; the quote branch (commented `// For yarn format`) and the
   `@npm:` replace are both Yarn-shaped. Remove them plus the stale doc
   example once pnpm fixtures confirm unreachability.
7. **Exact `', '` stage-1 delimiter,** matching Yarn's serializer and the
   stage 2 `parseMergedKey` rule.
8. **This document is canonical**, built on the identity-plan skeleton with
   the staged-plan's evidence and decision rationale folded in.
9. **Predecessor documents stay in place** in `.ymandrikov/plans/`,
   superseded but untouched.

Earlier resolutions carried forward from the predecessors:

- **Yarn 1 alias naming via descriptor-target extraction**, not `resolved`
  URL paths: registry-independent and reuses the same coordinate parser.
- **No public model expansion**: requested names and locators stay internal
  to the Yarn loader.

## Rejected alternatives

### Change `splitPackage`

Grouped keys, quoting, descriptors, locators, and aliases are Yarn
concepts. Adding them to the shared single-coordinate helper would widen
its contract and couple pnpm to Yarn serialization.

### Silent exclusion of ambiguous entries

Deterministic but hides a real change from a review tool; replaced by
exclude-with-warning.

### Keep the first descriptor as the long-term identity

The first descriptor is sufficient only when every descriptor means the
same package. Serialization order is not an identity guarantee and cannot
resolve aliases or conflicting targets. Acceptable only as stage 1's
interim mechanism, where it changes behavior solely on same-ident keys.

### Build aggregation unconditionally

The duplicate-identity claims were unverified in both predecessor plans.
Architecture built on an unconfirmed shape gets synthetic-only tests and
speculative invariants; the gate buys evidence for one afternoon of fixture
generation.

### Use the Berry locator for both name and directness

The locator supplies the installed package name but loses alias names
declared in `package.json`. Directness must remain a separate calculation
over requested names.

### Infer Yarn 1 package names from `resolved` URLs

Registry URL layouts are not part of Yarn descriptor semantics and vary
across custom registries. Npm alias targets are already encoded in
descriptors and can be parsed without URL heuristics.

### Emit one dependency per descriptor

Descriptors are requests, not installed package instances. Per-descriptor
emission duplicates public dependencies and creates duplicate `Change`
records (`calculateVersionDiff` has no dedup by id).

### `replaceAll('@npm:')`

Changes the symptom without restoring the descriptor boundary; a behavior
change to shared code with zero fixtures demanding it.

### Expand `Dependency` or `Change`

Requested names and raw locators are needed only while interpreting Yarn.
Exposing them across every loader and UI consumer would increase coupling
without improving the current issue's output.

## Scope boundaries

This design does not:

- expose requested aliases or locators publicly;
- redesign npm or pnpm dependency identity;
- detect a change whose only difference is direct versus transitive status;
- change npm tarball URL construction;
- change git fetching or version selection;
- recover package identity from arbitrary registry URLs;
- add dependencies;
- change lint configuration;
- surface loader warnings in the web UI (stderr only, for now).
