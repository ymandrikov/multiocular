# Overengineering Analysis: Yarn Lockfile Identity Plan

## Verdict

The plan is materially overengineered, although its central model is sound.

Separating requested names from resolved names is necessary: Yarn formally distinguishes descriptors from locators, and lockfile packages carry locator identity. Keeping this logic out of `splitPackage` is also the right boundary.

## Findings

### 1. High — the pure helper test suite violates repository instructions

The plan creates 13 direct tests for `resolveYarnIdentity` starting at `.ymandrikov/to-execute/ef9a6b53cbd8.md:358`. But `server/AGENTS.md:3` explicitly says to use integration tests and not test small helpers.

Remove `server/test/yarn-identity.test.ts`. Cover identity behavior through `yarn.load()` fixtures in `server/test/yarn.test.ts`. The pure module can remain if it genuinely improves production organization.

### 2. High — identity extraction has become a partial Yarn grammar validator

The proposed parser validates package-name syntax, quoting, group separators, empty references, aliases, unknown protocols, and embedded quotes (`ef9a6b53cbd8.md:207` and `ef9a6b53cbd8.md:564`).

That is broader than the loader needs and still is not a complete Yarn parser—for example, it rejects every embedded quote without implementing escaping and splits groups before parsing quoting. This combines extra code with a future-compatibility risk.

Prefer minimal structural parsing:

- Split emitted grouped descriptors.
- Find the outer `@` separator after an optional scope.
- Interpret `npm:` aliases.
- Parse the Berry locator's outer identity.
- Validate only the final name that will reach downstream npm loading.

### 3. Medium — excessive branding adds type ceremony without proportional safety

The design introduces seven brands plus six unchecked cast helpers: descriptor, ident, reference, target, requested name, resolved name, and aggregation key (`ef9a6b53cbd8.md:164` and `ef9a6b53cbd8.md:564`).

The repository requires branded types, but it does not require every temporary substring to have a distinct brand. Keep brands where values cross meaningful boundaries—primarily requested name, resolved name, and possibly the aggregation key. Plain strings inside one parsing function are sufficient.

### 4. Medium — the aggregation requirement is valid, but its model is generalized unnecessarily

Per-lockfile aggregation follows the stated contract. However, `YarnResolution.type` and the type component of its key are dead generality: every Yarn result is always `'npm'` (`ef9a6b53cbd8.md:1524`).

A per-file map keyed by `(name, version)` with `{ name, version, requested: Set<...> }` is enough. Construction can continue to hardcode `type: 'npm'`.

### 5. Medium — the test matrix repeats behavior at two layers

The plan contains 26 explicit tests, many duplicated between pure-parser and loader coverage: locator authority, fallback, malformed groups, aliases, and scoped keys. Several also pin exact internal error prose.

Loader-level tests can cover the same contract with fewer fixtures—roughly 8–12 focused cases—while allowing internal parsing to be refactored.

### 6. Low — Task 2 Step 0 is pure ceremony

The external Yarn 1 experiment explicitly cannot change the design (`ef9a6b53cbd8.md:1293`). Delete it. A step whose possible outcomes have no effect is not part of an executable plan.

### 7. Low — the plan document itself is over-specified

At 1,797 lines, it repeats the contract as invariants, architecture, data flow, failure semantics, test matrices, complete code, acceptance criteria, risks, and rejected alternatives. The planned implementation is likely shorter than the plan.

A compact plan should retain:

- Requested-versus-resolved identity rules.
- Locator authority and fallback policy.
- Per-lockfile aggregation contract.
- Loader-level regression scenarios.
- Required verification commands.

## Recommended Shape

Keep `yarn-identity.ts`, but reduce it to a small structural parser with one discriminated result and only boundary-level brands. Integrate it once with per-lockfile aggregation, test entirely through `yarn.load()`, and remove the speculative/manual steps and duplicated test layers.
